Файл: pub_server-protocol.json
Назначение: пример ПУБЛИЧНОГО формата серверных сообщений.
Используется как шаблон. Реальные значения и логика задаются в private/protocol/.

Структура:
  wrappers.<name>.t — тип серверного сообщения (строка, которая попадёт в ответ как есть).
  wrappers.<name>.p — тело сообщения (payload).

  hello
  ------------------------------------------------------------
  Ответ сервера на приветствие клиента / установку сессии.

  Пример:
    "hello": {
      "t": "SERVER_HELLO_RESPONSE",
      "p": {
        "session": {
          "field": "EXAMPLE_SESSION_ID"
        }
      }
    }

  Поле t:
    • "t": "SERVER_HELLO_RESPONSE" — тип приветственного сообщения сервера.
    • Строка задаётся в этом файле и без изменений возвращается клиенту.

  Поле p:
    • p.session.field — имя поля, в которое сервер положит сгенерированный sessionId.

  Как это работает:
    1) Node-сервер сам генерирует sessionId при подключении.
    2) Берёт значение из p.session.field (например, "EXAMPLE_SESSION_ID").
    3) В реальном ответе убирает объект session.
    4) Создаёт в p новое поле с этим именем и записывает туда sessionId.

  Пример при field = "EXAMPLE_SESSION_ID":
    Конфиг:
      "session": { "field": "EXAMPLE_SESSION_ID" }
    Реальный ответ:
      "p": { "EXAMPLE_SESSION_ID": "<сгенерированный sessionId>" }

  Таким образом, изменяя строку в p.session.field, вы управляете тем,
  как именно поле с идентификатором сессии будет называться в ответе сервера.


   subscribe
   ------------------------------------------------------------
   Ответ сервера на подписку клиента (первичное состояние стакана).

   Пример:
     "subscribe": {
       "t": "SERVER_SUBSCRIBE_RESPONSE",
       "p": {
         "ob": [
           {
             "i": "INSTRUMENT_CODE",
             "ask": [],
             "bid": []
           }
         ]
       }
     }

   Поле t:
     • "t": "SERVER_SUBSCRIBE_RESPONSE" — тип сообщения с начальным стаканом.
     • Строка задаётся в этом файле и возвращается клиенту без изменений.

   Поле p:
     • p.ob        — массив стаканов (обычно один объект).
     • p.ob[].i    — инструмент, который сервер поставит в реальном ответе.
     • p.ob[].ask/bid - форма ask/bid описана в публичных orderbook-файлах (pub_subscribe_*.json и orderbook_comments.txt);

   Как сервер выбирает значение i:
     • Режим выбирается параметром instrumentSource в файле setting/setting.json.
       Допустимые значения: "client" или "fixed".

     1) instrumentSource = "client"
          Сервер использует инструмент из клиентского запроса на подписку.
          То есть именно клиент определяет значение i.

     2) instrumentSource = "fixed"
          Сервер игнорирует инструмент, который прислал клиент.
          В этом режиме сервер использует инструмент,
          указанный в собственных приватных настройках (private protocol) "i":
            setting/private/protocol/priv_server-protocol.json

   unsubscribe
   ------------------------------------------------------------
   Ответ сервера на отписку клиента.

      Пример:
        "unsubscribe": {
          "t": "SERVER_UNSUBSCRIBE_RESPONSE"
        }

      Поле t:
        • "t": "SERVER_UNSUBSCRIBE_RESPONSE" — тип сообщения об успешной отписке.
        • Строка задаётся в этом файле и без изменений возвращается клиенту.

       quote
       ------------------------------------------------------------
       Ответ сервера с обновлениями котировок / стакана.

       Пример:
         "quote": {
           "t": "SERVER_QUOTE_RESPONSE",
           "p": []
         }

       Поле t:
         • "t": "SERVER_QUOTE_RESPONSE" — тип сообщения с обновлениями.
         • Строка задаётся в этом файле и возвращается клиенту без изменений.

       Поле p:
         • p — массив обновлений.
         • Каждый элемент массива представляет одно изменение в стакане
